# AGENTS.md — LogAnalyzer (Windows Event Log Analyzer)

このリポジトリは WPF (.NET 8) で作る Windowsイベントログ解析ツールです。
Codex エージェントはこのファイルの指示に従って作業してください。

---

## 1. 作業範囲（Allowed Scope）

- 作業対象は **このリポジトリのルート（E:\Dev\LogAnalyzer）配下のみ**。
- ルート配下のファイル・フォルダは **編集 / 追加 / 削除**してよい。
- ただし、生成物フォルダ（bin/obj）や一時ファイルは原則コミット対象外のため、内容の直接編集は不要。

---

## 2. 目的（Project Goals）

- Windowsイベントログ（主に System / Application）を期間指定で収集・一次解析する。
- 出力は `events.jsonl`（正） / `events.csv` / `summary.md` を基本とする。
- UI（WPF）は必須。CLI単体にはしない。
- 将来、リモート収集（WinRM/PowerShell Remoting）へ拡張できる構造にする。

---

## 3. 開発方針（Principles）

- **小さく変更し、ビルドが通る状態を保つ。**
- UIはまず「動く骨格」→「配線」→「収集」→「解析」の順に段階投入する。
- 収集（Collector）と解析（Analyzer）はクラス分離し、UI（View）にロジックを密集させない。
  - ただし初期は code-behind でもよい。後で分離しやすい形を意識する。

---

## 4. やってよいこと（Do）

- 既存WPFプロジェクトのファイル編集（XAML / .cs / .csproj）
- 新規クラス追加（例：`Models/`, `Services/`, `Core/` など）
- XAMLの追加・レイアウト改善（Menu/Toolbar/DataGrid/StatusBar 等）
- ボタンやメニューのイベント配線
- ログ取得・正規化・集計・ファイル出力の実装
- 例外処理・ログ（Run Log）・進捗表示・キャンセル対応（CancellationToken）の実装
- `README.md` の整備（使用方法・権限・注意事項）

---

## 5. やらないこと（Don't）

- **ワークスペース外（E:\Dev\LogAnalyzer 以外）を変更しない。**
- Visual Studio のGUIデザイナを「操作する前提」の手順にしない（XAMLを編集すること）。
- 既存の `x:Name` を勝手に変更しない（UI配線が壊れるため）。
- `bin/` `obj/` などビルド生成物を編集・追加しない（通常は.gitignore対象）。
- 依存関係の大量追加（NuGetパッケージをむやみに増やさない）。
  - 追加が必要な場合は「理由」「代替案」「影響範囲」を変更内容に明記する。

---

## 6. UIの前提（UI Contracts）

- メイン画面は `MainWindow.xaml` を中心に構成する。
- 主要UI要素（例：実行ボタン、出力先、プレビュー、サマリ表示、Run Log 等）は
  **Name付きコントロールとして維持**し、後続タスクで参照できるようにする。
- UIフリーズを避けるため、長い処理は `async/await` を使う（UIスレッドで重い処理をしない）。

---

## 7. 権限と同意（Admin + Consent）

このアプリは **管理者権限で実行する想定**です。
イベントログへのアクセスとログ内容（PC名、アプリ名、ユーザー名等が含まれる可能性）を扱うため、
初回起動時に同意（利用許諾）を取得するフローを採用します。

- 同意がない場合はアプリを終了する
- 同意済みフラグを Settings 等に保存して、次回以降は省略

---

## 8. ビルドと動作確認（Build / Run）

可能なら以下でビルドが通る状態を維持する：

- `dotnet build` が成功
- WPF起動が成功（XAMLエラー無し）

※CIは未整備。変更のたびに手元で最小限の起動確認ができる形を優先する。

---

## 9. 作業の進め方（Work Style）

- 1タスクは「UI追加」「同意画面追加」「収集実装」など **小さく区切る**。
- 変更内容は、影響ファイル一覧と、確認方法（どこを押すと何が起きるか）を簡潔にまとめる。
- 不明点があれば、推測で大改造せず、現状の構造を読んだ上で最小の改善を行う。
